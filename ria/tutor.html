Tutor Page & Week 7 Feedback

For the feedback week, I never had an anything prepared as I was still unsure of how I wanted to approach the project. Instead, I broke down the Specific Requirements individually and asked for feedback on some ideas to get a better understanding of the tasks and to then start preparing a more in depth idea on how to approach each of them.

General Requirements

I started by looking over the general requirements, but as these where mostly aesthetics, I would be leaving them until I started building the RIA website. Despite that, I started working on some website ideas where I came up with a starting point: screenshot-1.pdf

From here, I kept working on trying to get the basic website foundations down, using the World Wide Web as my scrapbook… Working my way through ideas and my feedback, including the general feedback PDF, I started getting comfortable with the now current one, whilst keeping the requirements and general feedback under constant careful thought.

Comparing and contrasting the different stages of the progression can be viewed as I changed ideas following these general requirements and evaluating the situation against each of the feedbacks: versions.php

Now that my site is complete, I can now evaluate the General Requirements…

Attractive Layout, colour, and use of images

I believe that my site is attractive and has a good use of contrasting colour, which aids in readability, as well as doesn’t go overboard with images. All images are relevant and not oversized and they aid in the way they are intended for them.

Good contrast between background and text

The site uses contrasting colours intended to aid with readability, with enough margin between text, images and background throughout to distinguish between content, providing the user with an optimal experience.

Clear functionality from the users' perspective

The website follows current trends, and the master detail arrangement to aid in the understanding of how the site works. Any interactive elements are indicated or otherwise automatic.

Purposeful interactive elements which enhances the appearance and functionality

There are segments throughout the website that utilise modern trends of interactive content and elements that are not extreme or overboard, but provide an overall pleasant experience with the content being viewed.

Overall

As an overall, I feel that the general requirements have been fulfilled to a high standard, with a conscientious effort to detail and these general guidelines.

However, if I was to do this again, I would change a few things to further enhance the site. I would change the layout of the site; perhaps have the navigation going down the left hand side of the screen, with the content on the right, using 100% x 100% of the screen size, and time permitting, in a responsive manner.

Using the same idea from above, I would remove the slideshow, and have a background image behind the text, maybe with a transparent layer for contrast and readability.

Specific Requirements

Reading over the specifics of the RIA requirements, I had no idea on what I wanted to include to the interactive elements of the website, or how to achieve doing so.

So during feedback week, I investigated each of the specific requirements by asking what was needed to achieve them individually to accomplish the end goal. By the end of the Q&A, I had a list some ideas and began to find some sources and resources that I can relate to for reference for each of the requirements…

A clear use of HTML with jQuery together with any related JavaScript

This requirement is stating that the end result must contain usage of jQuery, HTML and JavaScript, which the RIA website has been built upon.

View of information including images in a master detail arrangement

Using knowledge gained from last year, I can apply the master detail approach to the RIA website.

The interface should make use of at least one each from EVENTS, EFFECTS, and WIDGETS.

Events I knew about – they are just that – something that happens when something interacts with something, such as the user clicking on a button or link, pressing buttons on the keyboard or scrolling down the page. Events can also run from the browser – such as on document ready, or upon loading the document.

Effects I also knew, these are small animations throughout the webpage that can include sliding containers, hiding and showing elements, fading elements in and out, etc.

Acknowledging the feedback on events and effects, I have added a variety of examples throughout the RIA website, such as click events, input change, blur, hover, document ready for events, and for effects, I have a variety of animations throughout such as "slide" effects, and "blinks".

However, not knowing what was meant by a widget, I was told they are little utilities that can be applied to your website, such as a date or colour pickers, popup boxes and alerts, including content accordions and other text styling display methods.

In response to this, I added popup dialog boxes, such as the JS confirms when clicking on an external link that will allow the user to confirm or decline the option to follow the link. Each pop up is dynamic in the sense that it shows the actual link that you will be taken too rather than a generic "yes or no" box model.

A jQuery AJAX load of dynamic changing information

I knew what AJAX was as I have worked with it previously, but being able to use jQuery was a new experience. I was able to try new things such as the jQuery load, and only using some content from the new imported document - as jQuery has the option to parse the file contents leaving only the content you need.

A jQuery plugin which organises an aspect of your site

I spent a good few minutes talking about what kind of plugin to create for the project, and the two that where suggested included making some form of CSS style changing and the second changing the fonts sizes such as an accessibility styled plugin.

However, I know enough about JS and jQuery to create and implement something with a bit more complexity. So I set myself a challenge to create a plugin that will work with on top of another, keeping in mind advice of not to overcomplicate it, and ensure that it is still relevant to the website.

Another substantial and innovative feature which you have investigated yourself

Whilst talking about what kind of features that can be added, we spoke about the possibility of an RSS feed, or something similar. I also mentioned if I can get something like Amazon to be able to search for products, and this was given the OK if I was able.

In the end, I went with the eBay developer service, which I struggled with at the start, as it wouldn’t work at all. However, with some perseverance and maybe some stubbornness I got the basic form of it to work.I then later went on to add an appropriate level of customisability whilst following the JavaScript and jQuery guidelines of the module.

Overall

As an overall, creating the aspects of the Specific Requirements was easy enough, taking the provided feedback on board and developing something with it, providing some complexity to learn some new and advanced methods of programming, including using libraries and frameworks.

Reflecting on the specific requirements, most of the time was spent on the plugin and the additional shop feature. Although a significant time was spent on this, I would have loved to continue adapting and playing with them, allowing a more user focused approach, such as a form where the user can generate the data for the charts as an example, or a more secure and limited / specific form for the eBay shop.

In terms of feedback and suggestions provided, I have tried to implement it to the best of my ability following key techniques discussed such professional look and conquering the empty look, not having too much white space, and not going overboard with over large images and adding interactivity throughout the site without making it pointless or useless.
